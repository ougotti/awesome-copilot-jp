## Fundamentals（基礎）

### Agents、Skills、Instructions とは何か

**元ページ**: [What are Agents, Skills, and Instructions](https://github.com/github/awesome-copilot/blob/main/website/src/content/docs/learning-hub/what-are-agents-skills-instructions.md)
**読了時間**: 約 7 分 | **最終更新**: 2026-02-26

GitHub Copilot をカスタマイズする 3 つの主要なカスタマイズ要素を解説します。

#### Agents（エージェント）

`*.agent.md` ファイルで定義する設定ファイルです。専門的なタスク（例：「Terraform エキスパート」「LaunchDarkly フラグ管理者」）や使用するツール・MCP サーバー、会話スタイルのガイドラインを記述します。VS Code で Issue を Copilot にアサインしたり「Agents」パネルを開くと選択でき、専門的なアシスタントに切り替えられます。

**こんなときに使う**:
- 深いツール統合を必要とする定型ワークフローがある
- MCP を通じてコマンド実行やコンテキスト取得を自動で行いたい
- セッション全体を通じたペルソナレベルのガードレールが必要

#### Skills（スキル）

再利用可能な機能をパッケージ化した自己完結型フォルダです。各スキルは `SKILL.md` ファイルと、参照ドキュメント・テンプレート・スクリプトなどのオプションアセットを含みます。VS Code Chat で `/コマンド` として呼び出したり、エージェントが自動で発見・呼び出すことができます。

古い `*.prompt.md` 形式と比べた主な優位点：
- **エージェント自動発見**: エージェントが適切なスキルを自動で見つけて呼び出せる
- **リッチなコンテキスト**: 参照ファイル・スクリプト・テンプレートを同梱できる
- **クロスプラットフォーム対応**: [Agent Skills 仕様](https://agentskills.io/home) による移植性

**こんなときに使う**:
- 繰り返しタスクの対応方法を標準化したい
- テンプレート・スキーマ・スクリプトなどのリソースが必要
- エージェントに自動で発見・実行させたい

#### Instructions（インストラクション）

`*.instructions.md` ファイルで定義する背景コンテキストです。Copilot が対象ファイルを操作する際に自動で読み込まれます。コーディング規約・フレームワーク固有のヒント・リポジトリ固有のルールなどを記述します。

**こんなときに使う**:
- 多くのセッションにまたがる永続的なガイダンスが必要
- アーキテクチャ上の決定やコンプライアンス要件を定式化したい
- コンテキストを手動で貼り付けなくても Copilot に把握させたいパターンがある

#### 3 つの要素の使い分け

1. **Instructions** が長期的なガードレールとして土台を作る
2. **Skills** がオンデマンドで豊富なリソース付きの再利用可能なワークフローを提供する
3. **Agents** が最も意見の強い振る舞いをし、ツールとインストラクションを一つのペルソナにまとめる

### Agents とサブエージェント

**元ページ**: [Agents and Subagents](https://github.com/github/awesome-copilot/blob/main/website/src/content/docs/learning-hub/agents-and-subagents.md)
**読了時間**: 約 9 分 | **最終更新**: 2026-04-02

メインエージェントとサブエージェントの役割分担を解説します。

#### メンタルモデル：プロジェクトリードと担当者

| 項目 | Agent（メインエージェント） | Subagent（サブエージェント） |
|-----|--------------------------|--------------------------|
| 開始方法 | ユーザーが選択 / ワークフロー設定 | 別のエージェント・オーケストレーターが起動 |
| 寿命 | メイン会話・セッション全体を通じて継続 | 委任されたタスクの間だけ一時的に存在 |
| コンテキスト | 会話全体の目標を保持 | 狭いプロンプトと独立したコンテキストを持つ |
| スコープ | タスク全体を調整 | 一つの集中した作業を実行 |
| 出力 | ユーザーと直接対話 | メインエージェントに結果を報告し、集約される |

#### サブエージェントが有効な場面

- 実装前のリサーチ
- メインスレッドを汚さずに複数アプローチを比較
- 並列レビュー（正確性・セキュリティ・アーキテクチャなど異なる視点で同時実行）
- 大きな作業を依存関係のある独立したトラックに分割
- 異なるモデルで各サブタスクを処理

#### VS Code でのサブエージェント設定

フロントマターに `agent` ツールと許可エージェントリストを宣言することでサブエージェントを起動できます：

```yaml
---
name: Feature Builder
tools: ['agent', 'read', 'search', 'edit']
agents: ['Planner', 'Implementer', 'Reviewer']
---
```

#### Copilot CLI での活用

CLI では `/fleet` コマンドがオーケストレーターとして機能し、大きな目標を複数のサブエージェントに分解・並列実行・結果統合します。

```text
/fleet Update the auth docs, refactor the auth service, and add related tests.
```

#### オーケストレーションパターン

- **コーディネーターとワーカー**: 1 つのエージェントがワークフローを管理し、専門の作業者（プランナー、実装者、レビュアー）に委任
- **多視点レビュー**: 正確性・セキュリティ・コード品質・アーキテクチャなど異なる観点から並列レビュー
- **リサーチ→実行**: 1 つのサブエージェントが情報収集し、別のサブエージェントがその情報を使って実装

> **注意**: VS Code の `handoffs` フロントマタープロパティは GitHub.com のクラウドエージェントでは現在無視されます。エディタ固有の機能に依存する場合は移植性に注意してください。

### Copilot のコンテキストを理解する

**元ページ**: [Understanding Copilot Context](https://github.com/github/awesome-copilot/blob/main/website/src/content/docs/learning-hub/understanding-copilot-context.md)
**読了時間**: 約 8 分 | **最終更新**: 2025-11-28

GitHub Copilot が提案を生成する際に利用するコンテキストの種類と最適化方法を解説します。

#### Copilot が「見る」もの

| コンテキスト | 説明 |
|------------|------|
| **開いているファイル** | エディタで開中のファイルの内容 |
| **カーソル位置** | カーソル周辺のコード（前後） |
| **関連ファイル** | インポート・参照・依存関係から特定されるファイル |
| **チャット履歴** | Copilot Chat での過去のやり取り |
| **ワークスペース構造** | ディレクトリ構成・設定ファイル・パターン |

#### 4 種類のコンテキスト

1. **エディタコンテキスト**: 現在開いているファイルと画面上のコード
2. **セマンティックコンテキスト**: 関数シグネチャ・型定義・インターフェース・クラス階層・インラインコメント
3. **会話コンテキスト**: Copilot Chat の過去メッセージ・質問・回答
4. **ワークスペースコンテキスト**: `package.json`・`tsconfig.json`・`.gitignore` などのプロジェクトレベル情報

#### トークン上限と優先順位

コンテキストが多い場合、Copilot は以下の優先順位で処理します：

1. カーソルに最も近いコード
2. 明示的に参照したファイル（チャットでの `#` メンション）
3. 最近変更したファイル
4. 現在のファイルが直接インポートするファイル

#### ベストプラクティス

- **関連ファイルを開いておく**: コンポーネントのテストファイル・ユーティリティ・型定義を開く
- **説明的な名前を使う**: `getData()` より `getUserProfile()` の方がコンテキストが豊か
- **チャットで `#` メンションを使う**: 特定ファイルを明示的に参照して精度を上げる
- **コメントで意図を説明する**: 複雑なロジックの「なぜ」をコメントに書く
- **ワークスペースを論理的に構成する**: 明確なディレクトリ構造がコンポーネント間の関係理解を助ける

### MCP サーバーを理解する

**元ページ**: [Understanding MCP Servers](https://github.com/github/awesome-copilot/blob/main/website/src/content/docs/learning-hub/understanding-mcp-servers.md)
**読了時間**: 約 8 分 | **最終更新**: 2026-04-16

Model Context Protocol（MCP）サーバーが GitHub Copilot をどのように外部ツール・データベース・API へ接続するかを解説します。

#### MCP とは

**Model Context Protocol（MCP）** は AI アシスタントを外部データソース・ツールへ接続するためのオープン標準です。MCP サーバーはローカルで動作し、Copilot が会話中に呼び出せる「ツール」を公開します。

```
GitHub Copilot  ←→  MCP サーバー  ←→  外部システム
                     （ブリッジ）        （DB、API など）
```

#### 組み込みツール vs MCP ツール

| 組み込みツール | 機能 |
|--------------|------|
| `codebase` | リポジトリ全体のコード検索・分析 |
| `terminal` | ターミナルでのシェルコマンド実行 |
| `edit` | ファイルの作成・編集 |
| `fetch` | HTTP リクエスト |
| `github` | GitHub API との連携 |

MCP ツールはこれらを補完し、データベースクエリ・コンテナ管理・エラーレポート取得・デザイントークン読み取りなど外部サービスとの連携を実現します。

#### 設定方法

ワークスペースのルートに `.mcp.json` または `.vscode/mcp.json` を作成します：

```json
{
  "servers": {
    "postgres": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-postgres"],
      "env": {
        "DATABASE_URL": "${input:databaseUrl}"
      }
    }
  }
}
```

CLI でのインタラクティブなインストール：

```bash
/mcp install
# または
/mcp install @modelcontextprotocol/server-postgres
```

#### セキュリティのポイント

- **フォルダ信頼確認後のみ** ワークスペースの MCP サーバーが起動
- 機密値には `${input:variableName}` を使いファイルにシークレットをハードコードしない
- 最小権限原則を適用（分析用エージェントには読み取り専用 DB 接続）
- 組織ポリシーでサードパーティ MCP サーバーを制限可能

#### ベストプラクティス

- 必要最小限のアクセス権のみ付与
- API キーや接続文字列はファイルに書かず環境変数経由で渡す
- プロジェクトで使用する MCP サーバーをドキュメント化
- 認証情報を含むファイルは `.gitignore` でバージョン管理から除外

### Copilot 設定の基礎

**元ページ**: [Copilot Configuration Basics](https://github.com/github/awesome-copilot/blob/main/website/src/content/docs/learning-hub/copilot-configuration-basics.md)
**読了時間**: 約 10 分 | **最終更新**: 2026-04-29

ユーザー・ワークスペース・リポジトリの各レベルで GitHub Copilot を設定し、AI 支援開発体験を最適化する方法を解説します。

#### 設定の 3 レベル

| レベル | スコープ | 管理者 | 用途 |
|-------|---------|--------|-----|
| **ユーザー設定** | 全プロジェクト共通 | 個人 | 個人の好み・デフォルト動作 |
| **ワークスペース設定** | 特定プロジェクト | チーム | プロジェクト固有の規約 |
| **リポジトリ設定** | リポジトリ全体 | チーム | 共有ルール・エージェント・スキル |

#### リポジトリレベルのカスタマイズ配置

```
.github/
├── copilot-setup-steps.yml   # コーディングエージェント環境セットアップ
├── copilot-instructions.md   # リポジトリ全体の Copilot 指示
├── agents/                   # カスタムエージェント定義
├── skills/                   # スキルフォルダ
├── hooks/                    # ライフサイクルフック
└── instructions/             # コーディング規約
```

#### `copilot-setup-steps.yml`

Copilot コーディングエージェントがリポジトリで作業する際の環境セットアップ手順を定義します。新しい開発者へのブートストラップ手順として考えると分かりやすいです。

#### `copilot-instructions.md`

リポジトリ全体に適用されるグローバルな指示を記述します。チームの規約・技術スタック・アーキテクチャの決定をここに書くと、すべての Copilot インタラクションに適用されます。

### カスタム Instructions の定義

**元ページ**: [Defining Custom Instructions](https://github.com/github/awesome-copilot/blob/main/website/src/content/docs/learning-hub/defining-custom-instructions.md)
**読了時間**: 約 8 分 | **最終更新**: 2026-03-22

永続的でコンテキスト対応のインストラクションを作成し、コードベース全体で GitHub Copilot を自動的にガイドする方法を解説します。

#### カスタム Instructions とは

`*.instructions.md` ファイルで定義する設定ファイルで、マッチするファイルを操作する際に Copilot が自動で読み込みます。Skills（明示的な呼び出しが必要）とは異なり、バックグラウンドで自動適用されます。

含まれる内容：
- コーディング規約（命名規則・フォーマットルール・スタイルガイド）
- フレームワーク固有のガイダンス
- アーキテクチャ上の決定（構造・デザインパターン・規約）
- コンプライアンス要件（セキュリティポリシー・規制制約）

#### 基本構造

```markdown
---
description: 'TypeScript React コンポーネントのコーディング規約'
applyTo: '**/*.tsx, **/*.ts'
---

# TypeScript React 開発

すべての Props に TypeScript インターフェースを使用した関数コンポーネントを使用します。

## 命名規則
- コンポーネントファイル: PascalCase (例: `UserProfile.tsx`)
- フックファイル: camelCase + `use` プレフィックス (例: `useAuth.ts`)
```

#### スコーピングのベストプラクティス

`applyTo` フィールドで対象ファイルを指定します（文字列またはグロブ配列どちらでも可）：

```yaml
applyTo: '**/*.ts, **/*.tsx'     # 文字列形式（カンマ区切り）
applyTo:                          # 配列形式
  - '**/*.ts'
  - '**/*.tsx'
```

#### よくある落とし穴

| ❌ やりがちなミス | ✅ 改善策 |
|----------------|---------|
| 「きれいなコードを書く」などの曖昧な指示 | 「20 行以上の関数は小さな関数に分割する」など具体的なパターンを提示 |
| ドキュメントページ全体を貼り付ける | 重要なパターンだけ抽出し、詳細ドキュメントへのリンクを記載 |
| 矛盾する Instructions | 明確なスコープで補完的な Instructions を設計 |
| 言語固有の Guidelines に `applyTo: '**'` を使う | `applyTo: '**/*.py'` など適切なスコープを設定 |

### 効果的な Skills の作成

**元ページ**: [Creating Effective Skills](https://github.com/github/awesome-copilot/blob/main/website/src/content/docs/learning-hub/creating-effective-skills.md)
**読了時間**: 約 9 分 | **最終更新**: 2026-02-26

チーム全体で一貫した結果をもたらす再利用可能・共有可能なスキルフォルダの作り方を解説します。

#### Skills の仕組み

Skills はフォルダで、`SKILL.md` ファイル＋オプションのバンドルアセット（参照ファイル・テンプレート・スクリプト）で構成されます。`SKILL.md` は以下を定義します：

- **name**: ケバブケースの識別子（`/コマンド` としても機能）
- **description**: エージェントがいつこのスキルを使うべきかを示す説明
- **instructions**: Copilot が実行する詳細なワークフロー
- **asset references**: バンドルされたテンプレート・スクリプトなどへのリンク

#### フォルダ構造の例

```
skills/
└── generate-tests/
    ├── SKILL.md
    ├── references/
    │   └── testing-patterns.md
    ├── templates/
    │   └── test-template.ts
    └── scripts/
        └── setup-test-env.sh
```

#### 効果的な `description` の書き方

`description` フィールドはエージェントによる自動発見に使われます。エージェントが「**いつ**」このスキルを使うべきかが分かる書き方をしましょう：

✅ **良い例**: `'Generate conventional commit messages by analyzing staged git changes and applying the Conventional Commits specification'`

❌ **悪い例**: `'Commit helper'`

#### リポジトリの実用例

- [**conventional-commit**](https://github.com/github/awesome-copilot/tree/main/skills/conventional-commit): ステージング済み変更からコミットメッセージを自動生成
- [**excalidraw-diagram-generator**](https://github.com/github/awesome-copilot/tree/main/skills/excalidraw-diagram-generator): スキーマドキュメント・テンプレート・スクリプトをバンドルした図生成スキル

#### ベストプラクティス

- 1 つのスキルは 1 つの目的に絞る
- 発見しやすいように説明にトリガーキーワードを含める
- 幻覚を減らすためにテンプレート・スキーマ・参照ドキュメントをバンドルする
- 特定プロジェクトのパスをハードコードせず汎用的に書く

### カスタム Agents の構築

**元ページ**: [Building Custom Agents](https://github.com/github/awesome-copilot/blob/main/website/src/content/docs/learning-hub/building-custom-agents.md)
**読了時間**: 約 10 分 | **最終更新**: 2026-02-26

特定のペルソナ・ツール統合・ドメイン知識を持つ専門的な GitHub Copilot エージェントの作成方法を解説します。

#### カスタム Agents とは

`*.agent.md` ファイルで定義する設定で、GitHub Copilot に特化したペルソナ・ツールアクセス・ドメイン知識を付与します。

**フロントマターの主要フィールド：**

```yaml
---
name: 'Security Reviewer'
description: 'OWASP 脆弱性・認証欠陥・サプライチェーンリスクをレビューするエキスパートセキュリティ監査者'
model: Claude Sonnet 4
tools: ['codebase', 'terminal', 'github']
---
```

| フィールド | 説明 |
|----------|------|
| `name` | エージェントの表示名 |
| `description` | エージェントピッカーに表示される説明 |
| `model` | エージェントを動かす AI モデル |
| `tools` | エージェントが使えるツール・MCP サーバーの配列 |

#### エージェントの設計パターン

1. **ドメインエキスパート**: 特定技術への深い知識（例: Terraform + Azure 専門家）
2. **ワークフローオートメーター**: 多ステップのプロセスを実行（例: リリースマネージャー）
3. **品質ゲート**: 標準を強制する（例: アクセシビリティ監査者）

#### モデル選択の目安

| シナリオ | 推奨モデル |
|---------|-----------|
| 複雑な推論・セキュリティレビュー | Claude Sonnet 4 以上 |
| コード生成・リファクタリング | GPT-4.1 |
| クイック分析・シンプルなタスク | Claude Haiku / GPT-4.1-mini |
| 大規模コードベースの理解 | 大きなコンテキストウィンドウのモデル |

#### リポジトリ構成

```
.github/
└── agents/
    ├── security-reviewer.agent.md
    ├── api-designer.agent.md
    ├── terraform-expert.agent.md
    └── release-manager.agent.md
```

エージェントは集中させること（1 ファイル = 1 ペルソナ）。多くのことをやろうとするエージェントは、複数のエージェントに分割するか、共通タスクをスキルとして抽出します。

### プラグインのインストールと活用

**元ページ**: [Installing and Using Plugins](https://github.com/github/awesome-copilot/blob/main/website/src/content/docs/learning-hub/installing-and-using-plugins.md)
**読了時間**: 約 8 分 | **最終更新**: 2026-04-27

GitHub Copilot CLI を再利用可能なエージェント・スキル・フック・統合機能で拡張するプラグインを見つけ、インストール、管理する方法を解説します。

#### プラグインの構成要素

| コンポーネント | 役割 | ファイル場所 |
|-------------|------|------------|
| カスタムエージェント | 専門的な AI アシスタント | `agents/*.agent.md` |
| スキル | 個別の呼び出し可能な機能 | `skills/*/SKILL.md` |
| フック | エージェント動作をインターセプトするイベントハンドラー | `hooks.json` / `hooks/` |
| MCP サーバー | 外部ツール向け MCP 統合 | `.mcp.json` / `.github/mcp.json` |
| LSP サーバー | Language Server Protocol 統合 | `lsp.json` / `.github/lsp.json` |

#### マーケットプレイス

デフォルトで 2 つのマーケットプレイスが登録済みです（設定不要）：
- **`copilot-plugins`**: GitHub Copilot 公式プラグイン
- **`awesome-copilot`**: このリポジトリのコミュニティ貢献プラグイン

#### プラグインのインストール

```bash
# CLI から
copilot plugin install database-data-management@awesome-copilot

# インタラクティブセッションから
/plugin install database-data-management@awesome-copilot
```

VS Code では拡張機能検索で `@agentPlugins` と入力するか、コマンドパレットから「Chat: Plugins」を実行します。

#### 管理コマンド

```bash
copilot plugin list                          # インストール済みプラグイン一覧
copilot plugin update my-plugin             # プラグインを最新版に更新
copilot plugin marketplace update           # マーケットプレイスカタログを更新
copilot plugin uninstall my-plugin          # プラグインを削除
copilot --plugin-dir /path/to/my-plugin     # ローカルディレクトリからロード
```

#### チームでのマーケットプレイス共有

`.github/copilot-settings.json` に `extraKnownMarketplaces` を追加すると、チームメンバー全員が追加マーケットプレイスを自動取得できます：

```json
{
  "extraKnownMarketplaces": [
    {
      "name": "my-org-plugins",
      "source": "my-org/internal-plugins"
    }
  ]
}
```

### Hooks による自動化

**元ページ**: [Automating with Hooks](https://github.com/github/awesome-copilot/blob/main/website/src/content/docs/learning-hub/automating-with-hooks.md)
**読了時間**: 約 8 分 | **最終更新**: 2026-04-28

Copilot エージェントセッション中のライフサイクルイベント（フォーマット・リント・ガバナンスチェックなど）を自動化するためのフック活用方法を解説します。

#### Hooks とは

`.github/hooks/` に JSON ファイルとして格納するシェルコマンド・スクリプトで、Copilot エージェントセッション中のライフサイクルイベントに応じて自動実行されます。**確定的な自動化**（AI が「覚えているかどうか」に依存しないリント・フォーマット・ガバナンス走査）を提供します。

#### サポートされるイベント

| イベント | タイミング | 用途例 |
|---------|----------|--------|
| `sessionStart` | セッション開始時 | セッションログ記録・環境検証 |
| `sessionEnd` | セッション終了時 | 最終リント実行・通知送信 |
| `userPromptSubmitted` | ユーザープロンプト送信時 | プロンプトのコンプライアンス監査 |
| `preToolUse` | ツール実行前 | 危険なコマンドのブロック |
| `postToolUse` | ツール実行後 | コードフォーマット・検証 |
| `agentStop` | エージェントレスポンス完了時 | 最終チェック実行 |

#### 設定例

```json
{
  "hooks": [
    {
      "event": "postToolUse",
      "tool": "edit",
      "run": "prettier --write $EDITED_FILE"
    },
    {
      "event": "preToolUse",
      "tool": "terminal",
      "run": "scripts/block-dangerous-commands.sh"
    }
  ]
}
```

#### ベストプラクティス

- **必ず実行する必要がある操作**（フォーマット・リント）をフックにする
- `preToolUse` で危険な操作をブロック（破壊的ファイル操作・force push・DB DROP など）
- フックスクリプトはシンプルかつ高速に保つ
- フック失敗時の動作を明確にする（エラーでセッションを停止するか続行するか）

### Copilot コーディングエージェントの活用

**元ページ**: [Using the Copilot Coding Agent](https://github.com/github/awesome-copilot/blob/main/website/src/content/docs/learning-hub/using-copilot-coding-agent.md)
**読了時間**: 約 12 分 | **最終更新**: 2026-04-28

GitHub Issue を自律的に処理し、プルリクエストを作成・開発タスクを自動化する Copilot コーディングエージェントの使い方を解説します。

#### 仕組み

```
1. Issue を Copilot にアサイン（またはコメントで @mention）
         ↓
2. クラウド開発環境がスピンアップ
         ↓
3. Issue・Instructions・コードベースを読み込む
         ↓
4. ソリューションを計画・実装
         ↓
5. テストを実行して変更を検証
         ↓
6. レビュー用プルリクエストを作成
```

エージェントは独立したブランチ・隔離環境で動作し、コードのマージやデプロイは行いません。**必ず PR を作成し人間のレビューが必要**です。

#### 環境セットアップ：`copilot-setup-steps.yml`

コーディングエージェントがプロジェクトをセットアップする方法を定義します：

```yaml
# .github/copilot-setup-steps.yml
steps:
  - name: 依存関係のインストール
    run: npm ci

  - name: ビルド
    run: npm run build

  - name: テスト実行
    run: npm test
```

#### 作業の割り当て方

1. **Issue からの割り当て**: Issue に **Copilot をアサイン**
2. **コメントから**: `@copilot work on this` または具体的な指示をコメント
3. **カスタムエージェント指定**: `@copilot use the api-architect agent to implement this`

#### 効果的な Issue の書き方

```markdown
## 概要
/api/login エンドポイントにブルートフォース攻撃対策のレート制限を追加

## 要件
- IP アドレスごとに 15 分以内で 5 回に制限
- 上限超過時は HTTP 429 + Retry-After ヘッダーを返す
- 既存の Redis キャッシュを使用

## 受け入れ基準
- [ ] レート制限ミドルウェアが POST /api/login に適用されている
- [ ] テストが通常ログイン・制限ヒット・リセットをカバー

## コンテキスト
- レート制限ユーティリティ: src/middleware/rate-limiter.ts
```

**ポイント**: 具体的・既存コードを参照・「完了」の定義を明確に・スコープを適切に絞る

#### コーディングエージェントが得意なタスク

| ✅ 得意 | ❌ 不得意 |
|-------|---------|
| 明確な受け入れ基準がある機能実装 | チームでの議論が必要な設計上の意思決定 |
| 再現手順がある不具合修正 | 多数のファイルにまたがる大規模アーキテクチャ変更 |
| 既存コードへのテスト追加 | 開発環境にない外部システムへのアクセスが必要なタスク |
| 特定目標のリファクタリング | 明確なメトリクスがない性能最適化 |

#### リモートコントロール

実行中のコーディングエージェントセッションにローカル Copilot CLI から接続・誘導できます：

```bash
copilot --remote    # リモートアクセスを有効化してセッション開始
/remote on          # 既存セッション内でリモートコントロールを有効化
copilot --resume    # セッションピッカーから再接続
```

### Agentic Workflows

**元ページ**: [Agentic Workflows](https://github.com/github/awesome-copilot/blob/main/website/src/content/docs/learning-hub/agentic-workflows.md)
**読了時間**: 約 7 分 | **最終更新**: 2026-02-27

GitHub Agentic Workflows とは何か、Awesome Copilot コミュニティのワークフローの使い方、自分のワークフローを貢献する方法を解説します。

#### Agentic Workflows とは

GitHub Actions でコーディングエージェントを実行する AI 駆動のリポジトリ自動化です。マークダウンファイルに自然言語の指示を書くだけで、Issue トリアージ・日次レポート・コンプライアンスチェックなどをスケジュール・イベント・スラッシュコマンドで自動化できます。

**ワークフローファイルの構造：**

```markdown
---
name: "Daily Issues Report"
description: "オープン Issue の日次サマリーを生成"
on:
  schedule: daily on weekdays
permissions:
  contents: read
  issues: read
safe-outputs:
  create-issue:
    title-prefix: "[daily-report] "
    labels: [report]
---

## Daily Issues Report

チーム向けにオープン Issue の日次サマリーを作成します。
```

- **フロントマター**: トリガー・パーミッション・safe-outputs を宣言
- **本文**: エージェントが実行する自然言語の指示

#### 主な特徴

- 単一の `.md` ファイルで定義（YAML Actions 構文不要）
- スケジュール・リポジトリイベント・スラッシュコマンドでトリガー
- GitHub Actions 内で Copilot コーディングエージェントとして実行
- 最小権限・safe-outputs でセキュリティを確保
- `gh aw compile` で `.lock.yml` ファイルにコンパイル

#### 適用場面

| ユースケース | 例 |
|-----------|---|
| スケジュール済みレポート | 日次 Issue サマリー・週次組織ヘルスチェック |
| イベント駆動の自動化 | 新 Issue のトリアージ・PR の関連性チェック |
| スラッシュコマンド | Issue・PR への `/relevance-check` |
| コンプライアンスチェック | ライセンス監査・リリース準備レビュー |
| リポジトリメンテナンス | 古いリポジトリの特定・コントリビューター活動追跡 |

#### Awesome Copilot からワークフローをインストールする

```bash
# gh aw CLI 拡張機能のインストール
gh extension install github/gh-aw

# ワークフロー .md ファイルをリポジトリの .github/workflows/ にコピー後コンパイル
gh aw compile

# 検証のみ（ファイル生成なし）
gh aw compile --validate --no-emit workflows/my-workflow.md
```

#### コントリビューションのガイドライン

- セキュリティ最優先（最小権限・safe-outputs を使用）
- 明確・具体的な自然言語指示
- 説明的なファイル名（例: `daily-issues-report.md`）
- ローカルで `gh aw compile --validate` テスト後に提出

### カスタマイズ前後の比較例

**元ページ**: [Before/After Customization Examples](https://github.com/github/awesome-copilot/blob/main/website/src/content/docs/learning-hub/before-after-customization-examples.md)
**読了時間**: 約 12 分 | **最終更新**: 2025-12-12

カスタムエージェント・スキル・インストラクションが日常的な開発ワークフローをどのように変革するかを、具体的な Before/After で示します。

#### 例 1: API クライアントコード生成

**カスタマイズなし（Before）**:
- 汎用的な HTTP リクエストコード
- エラーハンドリング・リトライロジックなし
- チームの HTTP クライアントユーティリティを使用しない
- 型安全性・ロギングがない

**カスタム Instructions 適用後（After）**:
- チームの `HttpClient` クラスを自動使用
- 適切な TypeScript 型付き
- 構造化されたエラーハンドリングとロギング
- 環境変数ベースの設定
- 手動修正不要で規約に沿ったコードが生成

#### 例 2: テスト生成

**カスタマイズなし（Before）**:
- 汎用的なテスト構造
- フィクスチャやファクトリーなし
- モック戦略がない

**カスタム Skill 適用後（After）**:
- チームのテストユーティリティ・ファクトリーを自動使用
- 適切なセットアップ/ティアダウン
- ハッピーパス・エッジケース・エラーシナリオを網羅した包括的な assertions

#### 例 3: Terraform Infrastructure as Code

**カスタマイズなし（Before）**:
- Azure の命名規則に沿っていない
- タグ付けがない
- パブリックネットワークアクセスがブロックされていない
- 診断設定がない

**カスタムエージェント適用後（After）**:
- セキュアなデフォルト設定（暗号化・TLS 1.2+）
- Azure 命名規則・タグ戦略に準拠
- パブリックアクセスブロック・診断設定が含まれる
- 変数で設定可能・ハードコーディングなし

#### 例 4: コードレビューコメント

**手動レビュー（Before）**:
- 一貫性が低く時間がかかる
- 1 PR あたり 20〜30 分

**カスタム Skill 適用後（After）**:
- 構造化された一貫したフィードバック
- セキュリティ脆弱性を体系的に発見

### GitHub Copilot 用語集

**元ページ**: [GitHub Copilot Terminology Glossary](https://github.com/github/awesome-copilot/blob/main/website/src/content/docs/learning-hub/github-copilot-terminology-glossary.md)
**読了時間**: 約 8 分 | **最終更新**: 2026-04-02

GitHub Copilot のカスタマイズに関する主要用語の定義集です。Learning Hub の記事を読む際や、リポジトリを探索する際のクイックリファレンスとしてご活用ください。

#### 主要用語一覧

| 用語 | 定義 |
|-----|------|
| **Agent（エージェント）** | 特定の専門知識・ツール・振る舞いパターンを定義する設定ファイル（`*.agent.md`）。委任をサポートするプロダクトでは、主要なコーディネーターとして機能し、サブエージェントにタスクを委任する |
| **Subagent（サブエージェント）** | 別のエージェントまたはオーケストレーターが起動する一時的なタスク専用エージェント。メイン会話に戻らず結果を親エージェントに報告する |
| **Built-in Tool（組み込みツール）** | コード検索・ファイル編集・ターミナル実行・Web 検索など、追加設定不要の Copilot ネイティブ機能 |
| **Chat Mode** | **非推奨**。現在は「Agent」を使用 |
| **Collection** | awesome-copilot リポジトリ固有の概念。関連するスキル・インストラクション・エージェントをテーマ別にまとめた YAML ファイル（`*.collection.yml`） |
| **Front Matter** | マークダウンファイル先頭の `---` 区切り内の YAML メタデータ。ツールアクセス・モデル選択・スコープを制御 |
| **Handoff** | VS Code のカスタムエージェント用フロントマタープロパティ。GitHub.com クラウドエージェントでは現在無視される |
| **AGENTS.md** | 異なる AI コーディングツール間で共通に使えるポータブルな AI コーディング指示のファイル形式（[agents.md](https://agents.md/) 仕様） |
| **Instruction（インストラクション）** | マッチするファイルを操作する際に自動で読み込まれる永続的な背景コンテキスト（`*.instructions.md`）。コーディング規約・フレームワークヒント・リポジトリルールを含む |
| **Persona（ペルソナ）** | エージェントに定義されたアイデンティティ・トーン・振る舞いの特性 |
| **Prompt（プロンプト）** | **非推奨**。`*.prompt.md` 形式は Skills に置き換えられた |
| **Skill（スキル）** | `SKILL.md` とオプションのバンドルアセット（参照ドキュメント・テンプレート・スクリプト）を含む自己完結型フォルダ。[Agent Skills 仕様](https://agentskills.io/home) に準拠 |
| **MCP（Model Context Protocol）** | AI アシスタントを外部データソース・ツール・サービスに接続する標準プロトコル |
| **Hook（フック）** | Copilot エージェントセッション中のライフサイクルイベントに応じて自動実行されるシェルコマンド・スクリプト |
| **Coding Agent（コーディングエージェント）** | Issue に割り当てるとクラウド環境で自律的に作業し PR を作成する自律型 Copilot エージェント |
| **Plugin（プラグイン）** | エージェント・スキル・フック・MCP 設定・LSP 統合をバンドルした、マーケットプレイスからインストール可能なパッケージ |
| **Tools（ツール）** | Copilot がアクション実行や情報取得に使える機能群（組み込みツールと MCP ツールの 2 種類） |
